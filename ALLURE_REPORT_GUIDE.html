Opening a failed test

1Click Suites in the left nav
2Expand the spec file folder (e.g. specs/saucedemo/login.md)
3Click the red ✗ scenario to open the detail pane
4On the right pane: read Overview tab first (labels, params, timing)
5Switch to Test body tab to see step-by-step pass/fail trace
6Scroll to Attachments section — open observability JSON, screenshot, AI analysis

📌 Tip: the red error message shown at the top of a failed scenario is the exact Playwright exception — the same text the AI analyzer uses for classification. Read this first before looking at attachments.

Environment Panel

Environment Panel

Answers: "Which environment did this report come from?" — visible on the Overview page as the Environment widget.

What you see

FieldExample valueSource
EnvironmentsaucedemoENV_NAME in .properties file
BASE_URLhttps://www.saucedemo.comBASE_URL in .properties file
BrowserchromiumBROWSER in .properties file
HeadlesstrueHEADLESS in .properties file
AI_ProvideranthropicAI_PROVIDER (cfg)
DB_EnabledfalseDB_ENABLED (secret)
Run_IDlocal / 1234567890process.env.RUN_ID

✅ Values shown here always reflect the actual environment that ran — not the default. This is guaranteed by PLAYGAUGE_ENV and the cfg() resolver in EnvLoader.

How it is written

Written once at suite start by AllureSetup.writeEnvironment() called from the @BeforeSuite hook in hooks.ts:

// hooks.ts — @BeforeSuite
if (env.observabilityAttachAllure) {
  new AllureSetup(env).writeAll();
}

// AllureSetup.writeEnvironment() produces:
// allure-results/environment.properties
Environment  = saucedemo
BASE_URL     = https://www.saucedemo.com
Browser      = chromium
Headless     = true
AI_Provider  = anthropic
DB_Enabled   = false
Run_ID       = local

⚠️ If the Environment panel is empty or shows wrong values, the likely cause is running gauge directly without setting PLAYGAUGE_ENV. Always use npm run gauge:saucedemo (or the relevant gauge:* script).

Executors Panel

Executors Panel

Answers: "Who ran this report — a developer locally or GitHub Actions?" Shown as the Executor widget on the Overview page.

Local run — what it shows

{
  "name":       "Local",
  "type":       "local",
  "buildName":  "Local run — 22/05/2026, 10:34:21",
  "reportName": "Allure Report"
}

The timestamp tells you exactly when the local run happened, useful when comparing two reports from the same machine.

CI run — what it shows

{
  "name":       "GitHub Actions",
  "type":       "github",
  "url":        "https://github.com/org/repo",
  "buildOrder": 47,
  "buildName":  "Run #47",
  "buildUrl":   "https://github.com/.../runs/1234",
  "reportName": "Allure Report"
}

How to use the Executor panel

🔗

Click the build link

In a CI report the executor widget shows a clickable link to the exact GitHub Actions run that produced this report — jump directly to the logs.

🔢

Build order number

The buildOrder is the GitHub run number. Use it to cross-reference the Trend chart — each bar corresponds to one build order.

🖥️

Local vs CI distinction

When sharing a report with your team, the Executor panel immediately tells them whether the failure was reproduced on a fresh CI machine or only on your laptop.

Written by AllureSetup.writeExecutor() using GitHub Actions environment variables: GITHUB_RUN_NUMBER, GITHUB_REPOSITORY, GITHUB_RUN_ID, GITHUB_SERVER_URL. Automatically detected — no configuration needed.

Categories Panel

Categories Panel

Answers: "Why are tests failing — app bug, broken automation, or infra issue?" Only populated when tests actually fail.

The 3 failure buckets

🐛 Product Defects
Status: FAILED — error message matches assertion keywords
AssertionError · expect( · toHaveText · toBeVisible · toEqual · toContain
🔧 Test Issues
Status: FAILED / BROKEN — error message matches automation keywords
TimeoutError · locator.* · selector · Cannot read prop · is not a function
🌐 Infrastructure Issues
Status: FAILED / BROKEN — error message matches network keywords
ECONNREFUSED · ENOTFOUND · ERR_CONNECTION · net::ERR · ETIMEDOUT · Network request failed

📌 A test can appear in more than one category if its error message matches multiple regex patterns — e.g. a TimeoutError caused by a network failure shows in both Test Issues and Infrastructure Issues.

Why is my Categories panel empty?

✅ All tests passed

This is correct behaviour. Categories only classify failed tests. An empty panel is good news.

How to read a populated Categories panel

1️⃣

Click a category name

Expands to list every failed test that matched the pattern. Click any test to open its detail pane.

2️⃣

Read the matched error

Allure shows the specific part of the error message that triggered the category match — confirms why it was classified there.

3️⃣

Cross-check with AI Analysis attachment

Open the AI Analysis JSON attachment on the same test — it provides a confidence score and suggests the root cause. Allure categories are pattern-matched; AI reasoning is evidence-based.

⚠️ Categories are written to allure-results/categories.json by AllureSetup before the suite runs. If categories.json is missing, the panel will not appear even when tests fail.

Severity Labels

Severity Labels — Test Priority in Allure

Every scenario carries a Severity label that maps from the Gauge priority tag you add. Visible in the test detail pane under Labels.

Tag → Severity mapping

Gauge tag in specAllure SeverityMeaning
Tags: smoke, critical ● BLOCKER Stops the release. Fix before shipping.
Tags: regression, high ● CRITICAL Major impact. Fix in this sprint.
Tags: regression, medium ● NORMAL Moderate impact. Schedule for fix.
Tags: regression, low ● MINOR Low impact. Fix when time permits.
(no priority tag) ● NORMAL Default when no tag is provided.

✅ Severity tags are informational only — they never affect which tests are included or excluded from a run. They appear in Allure to help you triage which failures need immediate attention.

How to add severity to a scenario

# Login Tests
Tags: login, regression, saucedemo

## Successful login with standard user
Tags: smoke, critical          ← BLOCKER in Allure
* Open SauceDemo login page
* Login as "standard_user"
* Verify dashboard is displayed

## Login fails with wrong password
Tags: regression, high         ← CRITICAL in Allure
* Open SauceDemo login page
* Login as "standard_user" with wrong password
* Verify error message is shown

## Login page has all required elements
Tags: smoke, medium          ← NORMAL in Allure
* Open SauceDemo login page
* Verify all login elements are visible

Test detail pane — what each tab shows

📋

Overview tab

Labels (Severity, Suite, Feature, Tags), parameters (e.g. test.priority), test duration, host info. Check severity and tags here first.

🪜

Test body tab

Step-by-step trace. Each Gauge step shows ✅ / ❌ and its duration. The first red step is where the failure occurred — scroll to it immediately.

📎

Attachments section

At the bottom of the detail pane. Contains:
• observability-summary (JSON) — browser telemetry
• failure-screenshot (PNG) — browser state at failure
• ai-analysis (JSON, CI only) — AI classification result

📌 Quick triage flow: Overview tab → read Severity → Test body tab → find first red step → Attachments → open screenshot → open AI JSON for suggested fix.

Observability Attachment

Observability JSON Attachment

The observability-summary attachment is a JSON snapshot of everything the browser did during that scenario — captured automatically, no extra code needed in steps.

Full JSON structure

{
  "spec":              "specs/saucedemo/shopping.md",
  "test":              "Full checkout flow end-to-end",
  "module":            "Shopping Tests",
  "environment":       "saucedemo",
  "testStatus":        "passed",
  "failureMessage":    "",

  // ── Observability counts ──────────────────
  "consoleErrors":     0,
  "consoleWarnings":   2,
  "networkFailures":   0,
  "slowApiCalls":      1,
  "uncaughtExceptions":0,

  // ── Severity score ────────────────────────
  "severity":          "LOW",
  "severityScore":     4,

  // ── Rule-based hint for AI ────────────────
  "ruleCategoryHint":  "none",

  // ── API edges (calls made) ────────────────
  "apiEdges": [
    {
      "url":      "/api/v2/products",
      "method":  "GET",
      "status":  200,
      "duration":2340,          ← slow!
      "isSlow":  true
    }
  ]
}

Field-by-field guide

FieldWhat it tells you
consoleErrorsJavaScript errors from page.on('console') — a non-zero value usually indicates a front-end bug
consoleWarningsDeprecation warnings, unexpected states — low priority but worth noting
networkFailuresFailed XHR/fetch calls (page.on('requestfailed')) — strong signal for infra or API issues
slowApiCallsAPI calls that exceeded SLOW_API_THRESHOLD_MS (default 2000 ms) — performance signal
uncaughtExceptionsUnhandled JS exceptions (page.on('pageerror')) — highest severity signal
severityComputed: NONE / LOW / MEDIUM / HIGH based on weighted score
severityScoreRaw score: 5×uncaught + 4×netFail + 3×consErr + 2×slowApi + 1×warnings
ruleCategoryHintPre-classification hint fed to AI: flaky / environment / none
apiEdgesEvery API call: URL, method, HTTP status, duration ms, whether it was slow

💡 When a test fails: check networkFailures > 0 first — this immediately points to an API or infra problem rather than an automation issue. Then check uncaughtExceptions — this points to a product bug.

AI Analysis Attachment

AI Analysis JSON Attachment

Available in CI only (when AI_API_KEY is set). The ai-analysis attachment gives you a structured root-cause diagnosis for every failed test.

Full JSON structure

{
  "testName":       "Full checkout flow end-to-end",
  "runId":          "12345678901",
  "aiProvider":     "anthropic",
  "aiModel":        "claude-3-5-sonnet-20241022",
  "analysedAt":     "2026-05-22T01:15:43Z",

  // ── Primary classification ──────────────
  "category":       "Product Issue",
  "confidence":     0.91,

  // ── Evidence-based explanation ──────────
  "reasoning": "Stack trace shows TimeoutError
    on locator('.checkout-complete')
    immediately after a 500 response
    from /api/v1/orders. Two console
    errors confirm a server-side failure,
    not a selector or timing issue.",

  // ── Concrete next steps ─────────────────
  "suggestions": [
    "Check /api/v1/orders server logs
      at this timestamp",
    "Verify the order payload schema
      matches the API contract",
    "Add a contract test for the
      /api/v1/orders endpoint"
  ]
}

Field-by-field guide

FieldHow to use it
categoryOne of 6 classes: Automation Script Issue, Flaky Issue, Network / Infrastructure Issue, Product Issue, Test Data Issue, Unclassified
confidence0–1 scale. Above 0.8 = high confidence, act on it. Below 0.6 = treat as a starting hint, verify manually.
reasoning2–4 sentences citing specific evidence: error text, HTTP status codes, console errors, historical data. Read this to understand why the AI classified it this way.
suggestions3 concrete, actionable steps. Ranked by priority — follow the first one first.

The 6 failure categories

CategorySignal pattern
🤖 Automation Script IssueSelector broken, stale element, wrong locator
⚡ Flaky IssueHistorically flaky, timing/race condition
🌐 Network / Infrastructure4xx/5xx, ECONNREFUSED, timeout
🐛 Product IssueApp JS error, correct assertion fails
📦 Test Data IssueMissing/stale data, auth failure from data state
❓ UnclassifiedInsufficient signal to determine cause
Screenshot Attachment

Failure Screenshot

A full-page screenshot of the browser at the exact moment the test failed. Captured automatically in @AfterScenario when the scenario result is failed.

When is it captured?

❌

On failure only

Screenshots are not taken for passing tests — this keeps allure-results/ small and avoids noise.

📁

Saved to two places

1. dist/screenshots/<scenario-name>-<timestamp>.png for local inspection.
2. Embedded as a base64 attachment inside the Allure result JSON for the report.

🤖

Used by AI Analyzer

The screenshot is sent to the AI as a vision image (Claude 3+, GPT-4o, Gemini 1.5+). The AI can describe what the browser showed — misaligned UI, error dialog, blank page — as part of its reasoning.

How to find it in the report

1Open the failed scenario in Suites panel
2Scroll to the Attachments section at the bottom
3Click the failure-screenshot thumbnail
4Image opens full-screen — inspect the browser state at point of failure

What to look for in the screenshot

What you seeLikely cause
Blank / white pageNavigation failed, route doesn't exist, session expired
Error dialog / toastAPI returned an error, form validation triggered
Login page (unexpected)Authentication failure, session timeout mid-test
Spinner still visibleSlow API (check slowApiCalls in observability JSON)
Correct page, wrong dataTest data issue or timing race — check consoleErrors
Expected element missingSelector broke, feature flag disabled, wrong environment

💡 Correlation tip: Read the screenshot, then immediately open the observability JSON. If the screenshot shows a spinner → look for slowApiCalls > 0. If it shows a blank page → look for networkFailures > 0 or uncaughtExceptions > 0.

⚠️ Screenshots are only captured on HEADLESS=true runs. Local runs with HEADLESS=false also capture screenshots on failure; however never commit HEADLESS=false — it crashes CI.

Trend Chart

Trend Chart — Tracking Pass/Fail Over Time

The Trend chart shows a stacked bar for each CI run — passed (green) / failed (red) / broken (yellow) counts side-by-side. This is how you spot a degrading test suite before it becomes a crisis.

How trend history is persisted (CI)

Run N: tests execute → allure-results/ created
↓ allure generate reads results + restores history from previous run
Run N: allure-report/history/ written (contains N-1 bars)
↓ CI uploads history as artifact allure-history (overwrite: true, 90 days retention)
Run N+1 starts → CI downloads allure-history artifact
↓ history placed at allure-results/history/ before report generation
Run N+1: Trend chart now shows N bars (all previous + current)

📌 The CI step uses dawidd6/action-download-artifact@v6 to restore and actions/upload-artifact@v4 with overwrite: true to save. The first-ever run skips the restore step gracefully (continue-on-error: true) — the trend starts from run #2 onwards.

How trend history works locally

Local Trend — two-run process

Run 1: npm run gauge:saucedemo → npm run allure:generate — creates allure-report/history/

Run 2: npm run gauge:saucedemo — AllureSetup.restoreHistory() in BeforeSuite copies allure-report/history/ → allure-results/history/

Run 2: npm run allure:generate — Trend now shows 2 bars.

Reading the Trend chart

PatternWhat it means
All bars 100% greenSuite is stable — good baseline
One red spike then recoversTransient failure (infra / flaky) — check that run's Categories
Red growing over runsProgressive regression — check what changed in the codebase
Alternating red/greenFlaky test — check historical data in AI analysis
Yellow (broken) barsUnexpected exceptions — automation or infra problem

⚠️ The Trend chart only works across runs, not within a single run. A fresh first run always shows a single bar with no history context.

Behaviors Panel

Behaviors Panel — BDD Hierarchy View

The Behaviors panel groups tests by Feature → Story → Test. In playgauge_framework this maps to Gauge spec path → spec name → scenario name.

How Gauge tags become Behaviors

Allure BDD conceptMaps to in playgauge_framework
FeatureRelative path of the spec file, e.g. specs/saucedemo/login.md
SuiteSpec heading (the # Title in the .md file)
Story / TestIndividual scenario name (the ## Scenario in the .md file)
TagsAll Gauge spec-level + scenario-level tags — shown in the Labels widget

When to use Behaviors vs Suites

🌿

Suites panel

Best for triage — shows the exact file that failed, mirrors the repo structure, makes it easy to jump to the code.

📖

Behaviors panel

Best for stakeholder reporting — groups tests by feature area (login, checkout, cart) regardless of which file they live in. Easier to read for non-engineers.

Timeline Panel — at a glance

The Timeline panel shows a Gantt-style swimlane of when each test ran and how long it took.

⏱️

Duration outliers

A scenario bar that is much longer than its neighbours is likely hitting a slow API or waiting for a timeout — check its observability JSON for slowApiCalls.

🔀

Parallelism check

When running tests in parallel, the timeline shows overlapping bars — each thread on its own swim lane. playgauge_framework runs Gauge tests sequentially by default (one browser context per scenario).

📐

Total wall-clock time

The rightmost bar end minus the leftmost bar start = total wall-clock time of the suite run. Compare across runs to detect performance regressions in the suite itself.

📌 The Timeline is especially useful when you suspect a test indirectly breaks another by leaving the browser in a bad state. Sequential runs without proper cleanup show as one long unbroken bar chain.

Triage Workflow

Quick Triage Workflow — Step by Step

Use this checklist every time a CI run has failures. Takes 2–5 minutes per failed test.

5-step failure triage

1
Overview — how many and which severity?
Look at the doughnut chart. Count failures. Check if any are BLOCKER or CRITICAL — fix those first.
2
Categories — what type of failure?
🐛 Product Defect = app bug → raise a bug ticket. 🔧 Test Issue = fix the selector. 🌐 Infra = check servers.
3
Suites → Test body — which step failed?
Open the failed scenario. Go to Test body tab. The first red step pinpoints the exact action that threw the exception.
4
Screenshot — what did the browser show?
Open the screenshot attachment. Blank page? Login screen? Spinner? Each visual maps to a different root cause (see Slide 10).
5
Observability + AI JSON — confirm and fix
Open observability JSON → check networkFailures, uncaughtExceptions, slowApiCalls. Then open AI JSON → read reasoning and follow the first suggestion.

Signal → root cause cheat-sheet

Signal you seeWhere to look firstLikely fix
networkFailures > 0Check apiEdges for the failing URL; check server logsAPI or infra issue — not automation
uncaughtExceptions > 0Check browser console output; screenshotJavaScript error in the app — product bug
slowApiCalls > 0Check apiEdges for slow URL + durationPerformance regression; increase threshold only if intended
consoleErrors > 0Check screenshot; is UI broken visually?Front-end rendering issue — product bug
TimeoutError in step bodyIs the selector valid? Did the page load?Automation issue (selector) or slow page (infra)
AssertionError in step bodyWhat was expected vs actual? Check screenshotProduct regression or wrong test data
Test alternates pass/failCheck Trend chart for zigzag; check ruleCategoryHintFlaky test — add wait or fix race condition

🎯 Goal: 80% of failures should be diagnosed in under 3 minutes using this workflow — without ever running the test locally or opening the app manually.
