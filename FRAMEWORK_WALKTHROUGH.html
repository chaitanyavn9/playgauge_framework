Why This Stack?

  • Gauge specs are readable by non-engineers — product managers can write acceptance criteria directly as test scenarios
  • Playwright is faster and more reliable than Selenium — built-in auto-wait, network interception, trace viewer
  • Component Model means selector changes in one place only — not scattered across 50 test files
  • Observability collector gives zero-effort diagnostics — no manual logging required
  • AI analysis reduces triage time from hours to minutes
  • Grafana gives team-wide visibility without everyone needing to run tests
Design Pattern

Component Model Architecture

The key difference between this framework and a traditional Page Object Model (POM).

❌ Traditional POM

// LoginPage.ts — locators scattered everywhere
class LoginPage {
  async login(user, pass) {
    await page.locator('#username').fill(user);
    await page.locator('#password').fill(pass);
    await page.locator('button.submit').click();
  }
}

When #username changes → update every page that uses it.

✅ Component Model

// SauceLoginPage.ts — composes components
class SauceLoginPage extends BasePage {
  usernameInput = new InputComponent(
    page, obs, { css: '[data-test="username"]' }
  );
  loginButton = new ButtonComponent(
    page, obs, { css: '[data-test="login-button"]' }
  );
}

Selector changes in InputComponent once → fixes everywhere.

BaseComponent

Abstract base with interact(), waitForVisible(), assertVisible(), auto-wires ObservabilityCollector

Concrete Components

InputComponent, ButtonComponent, DropdownComponent, DataTableComponent — each knows how to interact with that UI type

Page = Composer

Pages declare which components they contain. Steps call page methods. Pages call component methods. Components touch the browser.

Directory Guide

Where Everything Lives

📁

specs/ — Write BDD scenarios here

Gauge Markdown (.md) files. One folder per application or module. Non-engineers can read and edit these.

🔌

tests/step-implementations/ — Wire steps to pages

TypeScript @Step functions that call page methods. One file per module. No locators allowed here.

🧩

tests/components/ — Reusable UI widgets

form/, data/, feedback/, navigation/ sub-folders. Add new components here when a new UI pattern appears.

📄

tests/pages/ — Thin orchestrators

Page objects that declare which components they use. One page per screen. Zero raw locators.

🌿

env/ — Per-environment configuration

Gauge .properties files. One folder per environment (dev, staging, saucedemo, etc.). DB_ENABLED=false for local.

🎭

playwright-tests/ — Standalone specs

Pure Playwright .spec.ts files, no Gauge needed. Good for quick debugging or onboarding new team members.

📊

tests/db/ + tests/ai-analyzer/ — CI only

PostgreSQL schema, repository, and AI failure analyser. Only activated when DB_ENABLED=true.

🔭

tests/observability/ — Auto-telemetry

Never edit unless extending the telemetry schema. Runs automatically on every scenario via hooks.ts.

Writing Tests — Step 1

Writing Gauge Spec Files (.md)

Spec files live in specs/<module>/. Use plain English. Tags drive selective execution.

Anatomy of a spec file

# Module Title                   ← spec heading

Tags: saucedemo, regression     ← spec-level tags

## Scenario: Successful checkout  ← scenario heading
Tags: smoke                      ← scenario-level tag

* Open SauceDemo login page      ← step (must match @Step)
* Login as "standard_user"
* Add "Sauce Labs Backpack" to cart
* Proceed to checkout
* Complete order as "Jane" "Doe" "94107"

Parameterised steps

|-------------|------------|

Writing Tests — Step 2

Step Implementations

Live in tests/step-implementations/<module>/. Connect English steps to page methods. No locators allowed here.

Pattern

import { Step, DataStore } from 'gauge-ts';
import { Page } from 'playwright';
import { SauceLoginPage } from '../../pages/saucedemo/SauceLoginPage';
import { ObservabilityCollector } from '../../observability/ObservabilityCollector';

// ── helpers (same pattern in every module) ──
function getPage() { return DataStore.ScenarioDataStore.get('page') as Page; }
function getObs()  { return DataStore.ScenarioDataStore.get('obs') as ObservabilityCollector; }
function login()   { return new SauceLoginPage(getPage(), getObs()); }

// ── step functions ──
@Step('Open SauceDemo login page')
async function openSauceLogin(): Promise<void> {
  await login().open();
}

@Step('Login as <username>')
async function loginAs(username: string): Promise<void> {
  await login().loginAs(username, 'secret_sauce');
}

Rules for step implementations

  • One file per application module
  • Step text must exactly match the spec file wording
  • Always get page and obs from DataStore
  • Instantiate page objects inside helper functions (lazy, fresh per call)
  • Call page methods — never Playwright APIs directly
  • Parameters in step text use angle brackets: <username>
  • Don't import Playwright locators here
  • Don't call page.locator() here
  • Don't assert with raw expect() here — keep assertions in pages/components

💡 How DataStore works

hooks.ts creates a fresh Playwright page + ObservabilityCollector before every scenario and stores them in Gauge's ScenarioDataStore. Your step files read them back. This is how the browser is shared across all steps of one scenario.

Writing a new component

import { BaseComponent } from '../base/BaseComponent';

export class DatePickerComponent extends BaseComponent {

  async selectDate(date: string): Promise<void> {
    await this.interact(`select date ${date}`, async () => {
      await this.el().click();
      await this.page.locator(
        `[data-date="${date}"]`
      ).click();
    });
  }

  async assertSelectedDate(expected: string): Promise<void> {
    const val = await this.getAttribute('value');
    expect(val).toBe(expected);
  }
}

Interpreting the Allure Report

📊

Overview / Summary

Total pass/fail/skip counts, duration, environment info. The first page you share with management.

🏷️

Categories

Failures grouped by type (product defects, test defects, broken tests). Map directly to AI analysis categories.

🔥

Suites

Tests organised by spec file → scenario. Drill into any failed scenario to see its steps, screenshots, and observability data.

📈

Trends (Retries + Flakiness)

Shows which tests consistently pass, fail, or are flaky over the last N runs. Only meaningful after 3+ CI runs.

🗂️

Behaviours (BDD view)

Tests grouped by Feature → Story → Scenario — the BDD hierarchy from your Gauge spec tags.

⏱️

Timeline

Visual swimlane showing when each test ran and how long it took. Useful for parallelisation planning.

🔭

Observability Attachments (custom)

Every scenario has: severity score, console error list, network failure list, slow API calls, API dependency edges. Attached automatically by ObservabilityCollector.

📸

Screenshots

Auto-captured on failure. Attached to the Allure scenario. Scroll to the bottom of a failed test to see the browser state at point of failure.

🤖

AI Analysis JSON (CI only)

The AI Analyser output is attached to each failed scenario: category, confidence, root cause, suggested fix. Also generates a standalone HTML summary report.

Severity scoring

score = (5 × uncaughtExceptions)
      + (4 × networkFailures)
      + (3 × consoleErrors)
      + (2 × slowApiCalls)
      + (1 × consoleWarnings)

0      → NONE
1-4    → LOW
5-9    → MEDIUM
10+    → HIGH

Why Section 2 (error + stack) matters

Previously failureMessage was always empty — the error was silently discarded. Now @AfterScenario reads executionContext.currentScenario.failedStep.errorMessage and .stackTrace directly from Gauge, so the AI sees the exact Playwright exception that caused the failure. Triage time drops from 30 min to 2 min.

Database

PostgreSQL — What Gets Stored and Why

5 tables in the schema

TableStores
test_runsOne row per scenario: all FailureFeaturesV1 fields, screenshot_base64, AI analysis result, git metadata
api_callsEvery HTTP call made during a test: URL, method, status, duration, is_slow flag
console_signalsEvery browser console message: type, text, source, timestamp
failure_trendsAggregated failure counts per test per day — powers flaky detection
ai_analysis_runsEach AI analysis batch: model, token usage, categories produced, run timestamp

How data enables flaky detection

-- Tests that failed AND passed in last 7 days
SELECT test_name,
       SUM(pass_count) AS passes,
       SUM(fail_count) AS failures
FROM failure_trends
WHERE recorded_at > NOW() - '7 days'::interval
  AND pass_count > 0
  AND fail_count > 0
GROUP BY test_name;

How data retrieval works

The AI Analyzer calls TestRunRepository.getFailedRunsForBatch(runId) to fetch all failed tests in a CI run. It also calls getHistoricalStats(testName) to add historical context before building the AI prompt. This lets Claude distinguish a first-time failure from a recurring flaky test.

PostgreSQL datasource query (example)

-- Pass rate over last 30 days
SELECT
  date_trunc('day', created_at) AS time,
  ROUND(
    100.0 * COUNT(*) FILTER
      (WHERE test_status = 'passed')
    / COUNT(*), 1
  ) AS pass_rate
FROM test_runs
WHERE created_at > NOW() - '30 days'::interval
GROUP BY 1
ORDER BY 1;

💡 Who uses Grafana?

Grafana is for team leads and managers who want a live view of test health without opening Allure. Share the dashboard URL with the team — no login required if anonymous access is enabled on the datasource.

Dos and Don'ts

Dos and Don'ts

✅ DO

  • Write step text in plain English a PM can read
  • Keep pages as thin orchestrators — no raw locators in page files
  • Put all UI interactions inside a Component extending BaseComponent
  • Prefer { automationId } when the app has data-automation-id attributes
  • Tag every spec file AND every scenario
  • Put credentials only in env .properties files
  • Run smoke tests before raising a PR
  • Keep DB_ENABLED=false on local environments
  • Read ObservabilityCollector telemetry when diagnosing flaky tests
  • Add a new component type when a new UI pattern appears

❌ DON'T

  • Don't put page.locator() calls in step implementations
  • Don't create page methods that duplicate component behaviour
  • Don't commit .env files or credentials to Git
  • Don't skip tagging — untagged scenarios can't be targeted
  • Don't use page.waitForTimeout(5000) — use component assertions
  • Don't install PostgreSQL or Grafana locally
  • Don't run DB_ENABLED=true without running db:migrate first
  • Don't hardcode test data inside step implementations
  • Don't modify ObservabilityCollector without understanding the schema
  • Don't run full regression locally for a quick sanity check — use --tags smoke
